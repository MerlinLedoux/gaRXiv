# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project

gaRXiv is a personalized RAG (retrieval-augmented generation) system for scientific watch on arXiv: targeted retrieval by domain, author, and source, automated daily monitoring, and summaries of new publications.

Ingestion, chunking, embedding/vector-storage, retrieval, and generation are implemented: fetching arXiv metadata, downloading PDFs, parsing them into structured text, cleaning and splitting them into overlapping token chunks, embedding those chunks into a vector store (Qdrant in dev, Azure Database for PostgreSQL + pgvector planned for prod), searching them via hybrid (dense + lexical) retrieval with metadata filtering, reranking, and query transformation, then synthesizing a grounded, cited answer to a natural-language question from the top retrieved chunks.

## Commands

- Install deps: `uv sync`
- Run ingestion: `uv run garxiv ingest [--limit N] [--dry-run] [--config-path config.yaml]`
- Run chunking: `uv run garxiv chunk [--limit N] [--config-path config.yaml]`
- Run embedding (needs a reachable vector store — see below): `uv run garxiv embed [--limit N] [--config-path config.yaml]`
- Create/verify the vector store collection: `uv run garxiv vectorstore-init [--config-path config.yaml]`
- Search: `uv run garxiv search "<query>" [--top-k N] [--category ...] [--author ...] [--date-from YYYY-MM-DD] [--date-to YYYY-MM-DD] [--hybrid/--no-hybrid] [--rerank/--no-rerank] [--query-transform none|multi_query|hyde]`
- Ask a question (retrieval + LLM-generated grounded answer with citations, needs Ollama running — see below): `uv run garxiv ask "<question>" [--top-k N] [--category ...] [--author ...] [--date-from YYYY-MM-DD] [--date-to YYYY-MM-DD] [--hybrid/--no-hybrid] [--rerank/--no-rerank] [--query-transform none|multi_query|hyde]`
- Backfill the FTS5 lexical index for chunks inserted before it existed: `uv run garxiv fts-reindex [--config-path config.yaml]`
- Run tests (excludes `slow`/`integration`-marked tests by default): `uv run pytest`
- Run a single test: `uv run pytest tests/test_pipeline.py::test_second_run_skips_already_processed_documents`
- Run the slow tests (loads the real BGE/reranker models) or integration tests (needs Qdrant/Ollama): `uv run pytest -m slow`, `uv run pytest -m integration`
- Start the dev vector store: `docker-compose up -d` (requires Docker Desktop running)
- Start the dev LLM (needed for `garxiv ask`, and optionally for `--query-transform multi_query|hyde`): install [Ollama](https://ollama.com), `ollama pull llama3.2:3b`, then `ollama serve`

## Architecture

Entry point is `src/garxiv/cli.py` (Typer), which loads `config.yaml` (categories, followed authors, storage paths, chunking/embedding/vector-store settings — validated via `src/garxiv/config.py`, a pydantic model) and dispatches to the pipeline functions below.

Every stage is idempotent and driven by a `status` column, either on `documents` or on `chunks`, both in SQLite (`src/garxiv/db.py`):

1. **Fetch** (`ingestion/arxiv_client.py`) — queries the arXiv Atom API for the configured categories/authors, filtered by a per-query watermark date (`ingestion_runs` table) so repeated runs only pull newly submitted papers instead of rescanning the archive. Results are upserted via `db.upsert_discovered`, which is a no-op for documents already known at their current version, and only resets an existing document back to `discovered` status (and deletes any of its existing chunks) when a newer version appears.
2. **Download** (`ingestion/downloader.py`) — processes only rows with `status = 'discovered'`, skips if the PDF file already exists on disk, then marks `status = 'downloaded'`.
3. **Parse** (`ingestion/parser.py`) — processes only rows with `status = 'downloaded'`, uses PyMuPDF to extract text and a heuristic section breakdown (headings detected via font size/boldness relative to the document's median; each line also keeps its page number), writes a JSON file, then marks `status = 'parsed'`.
4. **Chunk** (`chunking/pipeline.py`, via `run_chunking`) — processes only rows with `status = 'parsed'`: cleans each document (`chunking/cleaner.py` — strips arXiv watermark lines, isolated page numbers, recurring headers/footers, merges hyphen-split words), then splits it into overlapping token windows per section (`chunking/chunker.py`, tiktoken `cl100k_base`, configurable target tokens/overlap/min chunk size, skips sections like References by title). Chunks are stored in the `chunks` table (FK to `documents.arxiv_id`) with `section_title`, `page_start`/`page_end`, and `token_count`; the parent document moves to `status = 'chunked'`.
5. **Embed** (`embeddings/pipeline.py`, via `run_embedding`) — processes only chunks with `status = 'pending_embedding'`, in batches: embeds via the configured `EmbeddingProvider` (`embeddings/base.py`; `local_bge` using `sentence-transformers`/BAAI bge-small in dev, `azure_openai` stubbed for prod), upserts into the configured `VectorStore` (`vectorstore/base.py`; `qdrant` in dev via `vectorstore/qdrant_store.py`, `azure_pgvector` stubbed for prod), then marks the chunk `status = 'embedded'`. The vector store collection name encodes the embedding provider and dimension (`vectorstore/factory.py:collection_name`), so switching models can never silently mix incompatible vectors — it just requires a full re-embed. Qdrant point ids are internal (deterministic UUIDs derived from `chunk_id`); every public interface (`VectorStore`, `VectorMatch`) always speaks in terms of `chunk_id`. Each chunk's text is also indexed into a SQLite FTS5 virtual table (`chunks_fts`) at insert time (`db.insert_chunks`/`delete_chunks_for_document` keep it in sync explicitly, no SQL triggers), used for lexical search.
6. **Search** (`retrieval/pipeline.py`, via `run_search`) — read-only, never touches the database's write path. Pipeline: optional query transformation → dense + lexical search per query variant → Reciprocal Rank Fusion → optional reranking.
   - *Query transformation* (`retrieval/query_transform.py`, off by default via `search.query_transform.mode`): `multi_query` asks an `LLMProvider` (`llm/base.py`; `ollama` in dev via `llm/ollama_provider.py` calling the local Ollama REST API, `azure_openai` stubbed for prod) for N rephrasings; `hyde` asks it for a hypothetical passage, which is embedded with `is_query=False` (as a document, not a query) — this is the project's first generative LLM call, distinct from the embedding encoders. If the LLM is unreachable, `LLMUnavailableError` is caught and the pipeline falls back to the original query alone.
   - *Dense search*: each query variant is embedded (`is_query=True`, or `False` for the HyDE passage) and searched against the `VectorStore`.
   - *Lexical search* (`search.hybrid_enabled`, on by default): each query variant is sanitized (`retrieval/fts.py:sanitize_query` — tokenizes and quotes to neutralize FTS5 operators) and searched via `db.search_fts` against `chunks_fts`, joined with `documents` for category/author/date filtering.
   - *Fusion* (`retrieval/fusion.py:reciprocal_rank_fusion`): combines every dense and lexical ranked list (across all query variants) in one pass using RRF (`k=60`), avoiding the need to normalize dense (cosine) and lexical (bm25) scores onto the same scale.
   - *Filtering*: `SearchFilters` (category/author/date range) is translated into Qdrant's filter DSL (`vectorstore/qdrant_store.py:_build_filter`, supports `{"any": [...]}` and `{"gte"/"lte": ...}` in addition to scalar equality) and into `db.search_fts`'s SQL `WHERE` clauses.
   - *Reranking* (`search.rerank.enabled`, off by default — downloads a 278M-parameter model on first use): a `Reranker` (`rerank/base.py`; `local_cross_encoder` using `sentence-transformers.CrossEncoder` with `BAAI/bge-reranker-base`) rescoring the fused candidate pool before truncating to `top_k`.
   - SQLite (via `db.get_chunks_by_ids`) remains the source of truth for chunk text — never read from the Qdrant payload.
7. **Generate** (`generation/pipeline.py`, via `run_generation`) — read-only; calls `run_search` (point 6) to retrieve the top `top_k` chunks, then feeds up to `generation.context_top_k` of them (each truncated to `generation.max_chars_per_chunk` characters as a context-size safety cap) to the configured `LLMProvider` — the same `llm:` config section used for query transformation — via a numbered-excerpt prompt (`generation/prompt.py`) that instructs the model to ground its answer strictly in the excerpts, cite them by number (`[1]`, `[2]`...), say plainly if they don't answer the question, and reply in the same language as the question. Returns `AnswerResult(answer: str, sources: list[SearchResult])`, where `sources` is exactly the (possibly `context_top_k`-truncated) list of chunks sent to the LLM, in the same order used for numbering — so citation `[i]` always maps to `sources[i-1]`. If `run_search` finds nothing, the LLM is never called and a canned "no relevant documents" answer is returned directly, avoiding a wasted call and hallucination risk. Unlike query transformation, there is no fallback on LLM failure: `LLMUnavailableError` propagates out of `run_generation` uncaught, and is only caught at the CLI layer (`garxiv ask`), which prints a clean error message instead of crashing.

This status-per-stage design is what makes the pipeline resumable after a crash and avoids redundant work (e.g. re-embedding an already-processed chunk).

Each ingestion stage function takes an injectable `requests.Session`, which is how tests (`tests/test_arxiv_client.py`, `tests/test_pipeline.py`) mock the arXiv API and PDF downloads via the `responses` library instead of hitting the network. `tests/test_parser.py` generates synthetic PDFs on the fly with PyMuPDF rather than committing binary fixtures. Embedding/vector-store/search tests use fakes (`FakeEmbeddingProvider`, `FakeVectorStore`, `FakeLLMProvider`, `FakeReranker`) or Qdrant's embedded `:memory:` mode to avoid touching real models or requiring Docker/Ollama; only `-m slow` and `-m integration` tests do. The generation stage's tests (`tests/test_generation_pipeline.py`) reuse the same `FakeLLMProvider` pattern but monkeypatch `garxiv.generation.pipeline.run_search` directly with canned results, rather than re-mocking retrieval's own embedding/vector-store dependencies.

Local data (SQLite DB, downloaded PDFs, parsed JSON) lives under `data/`, which is gitignored. The dev vector store (Qdrant) runs via `docker-compose.yml` and persists to a named Docker volume.
